*Security Engineering · Technical Deep-Dive*

Security compliance frameworks exist to answer one uncomfortable question: Can you prove your systems are secure? Not claim it — prove it with logs, alerts, and audit trails that hold up under third-party scrutiny. This document walks through a practical architecture combining Fleet (osquery management), OpenEDR (endpoint detection and response), and Wazuh (SIEM + compliance monitoring) into a cohesive stack that produces the evidence SOC 2, HIPAA, and PCI DSS auditors require.

---

## Why These Three Tools?

### Fleet — Query & Inventory Layer

Fleet is an open-source platform for managing osquery across all endpoints — servers, workstations, and cloud instances. Rather than pushing data, Fleet lets you query your infrastructure like a database. Scheduled queries run continuously and ship results to a central log store, giving auditors a queryable record of system state over time rather than a point-in-time snapshot.

### OpenEDR — Behavioral Detection Layer

OpenEDR provides continuous behavioral monitoring — watching for suspicious process trees, credential access patterns, lateral movement, and file system anomalies. It generates telemetry modeled on the MITRE ATT&CK framework, capturing the full story of what happened on an endpoint: which process spawned what child, what file was accessed, what network connection was established.

### Wazuh — SIEM & Compliance Engine

Wazuh is the integration layer. It ingests logs from virtually any source, applies correlation rules to detect threats and policy violations, and ships with out-of-the-box compliance decoders for SOC 2, HIPAA, PCI DSS, NIST 800-53, and GDPR. Its compliance dashboards map every event to specific control requirements, producing audit-ready reporting without weeks of manual work.

## Integration Setup

### Step 1: Configure Fleet Log Output

Set Fleet to write osquery results as JSON to a log file that can be picked up by Filebeat:

```yaml
# fleet.yml — result log configuration
logging:
  result:
    plugin: filesystem
    config:
      result_log_file: /var/log/fleet/osquery_results.log
  status:
    plugin: filesystem
    config:
      status_log_file: /var/log/fleet/osquery_status.log
```

### Step 2: Forward Fleet Logs to Wazuh via Filebeat

```yaml
# filebeat.yml
filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /var/log/fleet/osquery_results.log
    fields:
      source: fleet-osquery
      compliance: true

output.logstash:
  hosts: ["wazuh-manager:5044"]
  ssl.certificate_authorities: ["/etc/filebeat/wazuh-ca.pem"]
```

### Step 3: Ingest OpenEDR Telemetry via Wazuh Agent

Deploy the Wazuh agent alongside OpenEDR on each endpoint. Configure the agent to read OpenEDR's JSON telemetry directly:

```xml
<!-- ossec.conf — on each endpoint -->
<ossec_config>
  <localfile>
    <log_format>json</log_format>
    <location>C:\ProgramData\OpenEDR\Logs\*.json</location>
    <label key="source">openedr</label>
  </localfile>
  <!-- Windows Security Event Log for HIPAA audit trail -->
  <localfile>
    <log_format>eventchannel</log_format>
    <location>Security</location>
  </localfile>
</ossec_config>
```

### Step 4: Add Wazuh Rules for OpenEDR Events

Map OpenEDR's MITRE ATT&CK categories to compliance control groups so Wazuh tags events automatically:

```xml
<!-- openedr_rules.xml -->
<group name="openedr,mitre,">
  <rule id="91001" level="12">
    <decoded_as>json</decoded_as>
    <field name="source">openedr</field>
    <field name="threat.type">credential_access</field>
    <description>OpenEDR: Credential access attempt detected</description>
    <mitre><id>T1003</id></mitre>
    <group>pci_dss_10.6.1,hipaa_164.312.b,</group>
  </rule>
</group>
```

## Key Fleet Compliance Queries

Schedule these osquery queries in Fleet to continuously collect evidence. Results are forwarded to Wazuh automatically.

### Disk Encryption Status — HIPAA / SOC 2

Detects any unencrypted disks across all hosts:

```sql
SELECT de.name, de.encrypted, de.type, h.hostname
FROM disk_encryption de
JOIN system_info h
WHERE de.encrypted = 0;
```

### Local Admin Accounts — SOC 2 CC6 / PCI DSS 8

Lists all non-root privileged accounts:

```sql
SELECT u.username, u.uid, g.groupname
FROM users u
JOIN user_groups ug ON u.uid = ug.uid
JOIN groups g ON ug.gid = g.gid
WHERE g.groupname IN ('admin', 'sudo', 'wheel')
  AND u.username != 'root';
```

### Outdated Packages — PCI DSS 6 / SOC 2 CC7

Cross-references installed packages against a vulnerability list:

```sql
SELECT name, version, source
FROM deb_packages
WHERE name IN ('openssl', 'libssl-dev', 'curl')
ORDER BY name;
```

## Log Retention by Framework

Configure Wazuh's OpenSearch Index Lifecycle Management (ILM) to meet each framework's retention requirements:

| Tier | Duration | Purpose |
|---|---|---|
| Hot | 0–30 days | Full resolution, immediate query access for incident response |
| Warm | 30–365 days | Compressed, available within minutes — satisfies PCI DSS 12-month requirement |
| Cold/Archive | 1–6 years | Encrypted snapshots to S3-compatible storage — required for HIPAA's 6-year mandate |
| SIEM access logs | Ongoing | Record of who queried the SIEM and when — required for SOC 2 CC6.1 |

## Answering Auditor Questions

**"Show me your access control monitoring."**
Export the Wazuh SOC 2 CC6 dashboard. Fleet query history shows every host's user account state across the entire audit period.

**"Show me your incident detection capability."**
OpenEDR's MITRE ATT&CK coverage map combined with Wazuh rule statistics demonstrates active detection across the full attack surface.

**"Were there any security incidents during the audit period?"**
Wazuh's alert history provides a complete, timestamped, immutable record. Incidents present means your detection-and-response process worked as designed.

**"How do you detect unauthorized software changes?"**
Fleet's software inventory queries and Wazuh File Integrity Monitoring (FIM) provide a continuous record of what was installed, when, and on which host.

## Getting Started

Deploy in this order to reach a working compliance baseline as quickly as possible:

1. Deploy Wazuh Manager + Indexer + Dashboard *(min. 4-core, 8GB RAM for production)*
2. Install Wazuh agents on all in-scope endpoints
3. Stand up Fleet server and enroll endpoints with the osquery agent
4. Configure Fleet → Filebeat → Wazuh log pipeline
5. Enable Wazuh compliance modules: PCI DSS, HIPAA, NIST 800-53
6. Schedule Fleet compliance queries (disk encryption, user accounts, software inventory)
7. Configure alert routing to PagerDuty and Slack
8. Set ILM retention policies matching your longest compliance requirement

This architecture scales from a 50-endpoint startup pursuing SOC 2 Type I to a 10,000-endpoint enterprise maintaining PCI DSS certification. The open-source foundation keeps licensing costs near zero, and the modular design allows adding commercial EDR feeds, cloud security logs, or network monitoring without replacing the core pipeline.
